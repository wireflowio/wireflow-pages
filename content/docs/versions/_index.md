## 🔄 Wireflow 版本历史

Wireflow 致力于快速迭代和稳定发布。以下是我们主要版本的发布记录和重要更新。

开源软件版本号最常用且被广泛接受的定义规则是 语义化版本（Semantic Versioning, 简称 SemVer）。
wireflow 同样遵循这一版本定义规则：
SemVer 提供了一套清晰、一致的规则，帮助用户（开发者）理解版本号的变化意味着什么，从而预测升级带来的风险和兼容性变化。

## MAJOR（主版本号）
- 何时增加： 当进行了不兼容的 API 更改时。
- 含义： 升级到新的 MAJOR 版本需要用户进行代码修改，或者做额外的配置调整。这通常是一个重大的版本升级。
- 重置： 增加 MAJOR 版本号时，MINOR 和 PATCH 版本号必须重置为 0。
- 示例： 从 1.x.x 升级到 2.0.0。

## MINOR（次版本号）
- 何时增加： 当进行了向下兼容的功能性新增或改进时。
- 含义： MINOR 版本通常会新增功能，但不会破坏现有代码的兼容性。
- 重置： 增加 MINOR 版本号时，PATCH 版本号必须重置为 0。
- 示例： 从 1.2.x 升级到 1.3.0。
## PATCH（修订版本号）
- 何时增加： 当进行了向下兼容的 Bug 修复时。
- 含义： PATCH 版本只包含错误修复或非常小的非功能性改进，对用户代码没有影响。
- 重置： PATCH 版本号不变或递增。
- 示例： 从 1.2.1 升级到 1.2.2。

## 🏷️ 版本号的特殊扩展
SemVer 也允许在主版本号后添加标签，以表示预发布或构建元数据。 
1. 预发布版本（Pre-release）预发布版本号跟在 PATCH 版本号后面，用连字符（-）分隔，用于标识不稳定的、可能有 Bug 或正在测试的版本。
- 常见标签： alpha (A), beta (B), rc (Release Candidate)。
- 示例： 1.0.0-alpha.1 $\rightarrow$ 1.0.0-beta.1 $\rightarrow$ 1.0.0-rc.1。
- 规则： 预发布版本低于对应的正式版本。例如，1.0.0-alpha 低于 1.0.0。2. 构建元数据（Build Metadata）构建元数据跟在预发布版本或 PATCH 版本号后面，用加号（+）分隔，用于提供构建信息，例如提交哈希、构建日期等。示例： 1.0.0-alpha.1+20251130规则： 构建元数据不参与版本号的优先比较。

2. 构建元数据（Build Metadata）
   构建元数据跟在预发布版本或 PATCH 版本号后面，用加号（+）分隔，用于提供构建信息，例如提交哈希、构建日期等。
- 示例： 1.0.0-alpha.1+20251130
- 规则： 构建元数据不参与版本号的优先比较。
